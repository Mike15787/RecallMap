# 3. 模組規範

## 3.1 核心原則

每個模組必須遵守以下三條規則，缺一不可：

- **單一職責**：一個模組只做一件事，例如 `pdf_parser.py` 只負責解析 PDF
- **AI 呼叫例外**：`image_parser.py` 可直接使用 `GemmaClient` 進行視覺理解；`document_parser.py` 使用 markitdown 做文字萃取，**不得**呼叫 AI（markitdown 本身即可處理，不需要 LLM）；其餘 ingest parser 不得呼叫 AI
- **統一介面**：每個 ingest 模組對外只暴露一個 `process()` 函式，返回相同的 `DocumentChunk` 格式
- **獨立可測試**：每個模組必須能在不啟動整個應用的情況下單獨執行測試

## 3.2 統一資料格式 — DocumentChunk

所有輸入模組的輸出必須轉換為此格式，engine 層只接受此格式作為輸入。

```python
# backend/ingest/base.py
from dataclasses import dataclass
from enum import Enum
from typing import Optional

class SourceType(Enum):
    PDF         = "pdf"
    PPT         = "ppt"
    WORD        = "word"
    IMAGE       = "image"
    NOTION      = "notion"
    CHATGPT     = "chatgpt"
    GEMINI      = "gemini"

# SourceType.PDF / PPT / WORD 三種均由 document_parser.py 的 markitdown 處理。
# 型別保留原值是為了讓 engine 層可區分來源（出題策略可能因格式不同而差異）。
# 轉換後的 content 欄位統一是 Markdown 字串，engine 層無需知道底層用了什麼工具。

@dataclass
class DocumentChunk:
    content: str                  # 純文字內容（必填）
    source_type: SourceType       # 來源類型（必填）
    source_id: str                # 來源唯一識別碼（必填）
    metadata: dict                # 額外資訊，如頁碼、對話時間戳等
    is_conversation: bool = False # 是否為對話紀錄（影響盲點偵測邏輯）
    language: str = "zh-TW"       # 內容語言
```

## 3.3 ingest 模組介面規範

每個 parser 必須實作以下介面，不得有其他公開函式：

```python
# 正確範例
async def process(source: str | Path | dict) -> list[DocumentChunk]:
    """單一公開入口，所有邏輯在內部處理"""
    ...

# 錯誤範例 — 不得暴露內部函式
def _parse_page(page):   # 私有函式加底線，不對外暴露
    ...
def parse_text(text):    # 禁止：不符合統一介面
    ...
```

## 3.9 markitdown 整合規範（文件解析層）

> **依賴**：`pip install 'markitdown[all]'`（`pyproject.toml` 加入 `markitdown[all]` 於 dependencies）
> 

`document_parser.py` 是 markitdown 在本專案的**唯一整合點**，其他模組不得直接 import markitdown。

```python
# backend/ingest/document_parser.py
# RUNTIME: edge（純 CPU，本地執行，不呼叫任何 AI API）

from pathlib import Path
from markitdown import MarkItDown
from .base import DocumentChunk, SourceType

# 單例：MarkItDown 初始化有成本，模組載入時建立一次
_md = MarkItDown(enable_plugins=False)

# 支援的副檔名對應到 SourceType
_EXT_TO_SOURCE = {
    ".pdf":  SourceType.PDF,
    ".pptx": SourceType.PPT,
    ".ppt":  SourceType.PPT,
    ".docx": SourceType.WORD,
    ".doc":  SourceType.WORD,
}

async def process(source: str | Path) -> list[DocumentChunk]:
    """
    將 PDF / PPTX / DOCX 轉換為 DocumentChunk 列表。
    內部使用 markitdown 做 Markdown 萃取，不呼叫任何 AI。
    """
    path = Path(source)
    ext = path.suffix.lower()

    if ext not in _EXT_TO_SOURCE:
        raise ValueError(f"document_parser 不支援此副檔名：{ext}")

    source_type = _EXT_TO_SOURCE[ext]

    # markitdown 核心呼叫：同步，但在 async 環境中足夠輕量
    result = _md.convert(str(path))
    markdown_text = result.text_content

    if not markdown_text or not markdown_text.strip():
        raise ValueError(f"markitdown 無法從檔案萃取文字：{path.name}")

    return [
        DocumentChunk(
            content=markdown_text,
            source_type=source_type,
            source_id=str(path.resolve()),
            metadata={
                "filename": path.name,
                "extension": ext,
                "char_count": len(markdown_text),
            },
            is_conversation=False,
        )
    ]
```

### markitdown 的限制與對策

| 情況 | markitdown 行為 | 對策 |
| --- | --- | --- |
| 掃描版 PDF（無文字層） | 回傳空字串或極少文字 | 偵測 `char_count < 100` → 拋出明確錯誤，提示使用者改用「拍照上傳」走 `image_parser.py` |
| PPT 內嵌圖表或圖片 | 圖片內容被略過，僅保留文字 | metadata 加入 `has_image_content: bool` 標記，engine 層可據此提示使用者「此投影片含圖片，建議補充說明」 |
| 大型檔案（> 5 MB） | 轉換時間可能超過 3 秒 | 在 FastAPI route 層用 `run_in_executor` 包裹，避免 block event loop |
| 受密碼保護的 DOCX/PDF | markitdown 拋出例外 | `except Exception` → 回傳使用者友善訊息「這份文件有密碼保護，請先移除密碼再上傳」 |

### 在 FastAPI route 中使用

```python

    # 3. 清掉暫存
```

### convert_stream() 注意事項（最新版 breaking change）

markitdown 最新版 `convert_stream()` 只接受 **binary** file-like object（`bytes` / `io.BytesIO`），不再接受 `io.StringIO`。如果需要從記憶體 bytes 轉換（例如從上傳的 bytes 直接處理，不落磁碟），使用方式如下：

```python
import io
result = _md.convert_stream(
    io.BytesIO(file_bytes),
    stream_info={"extension": ".pdf"}  # 必須提供副檔名提示，否則無法判斷格式
)
```

目前 `document_parser.py` 採落磁碟路徑（`convert(path)`），較穩定；之後如需最佳化記憶體使用，可改用 `convert_stream()`。

## 3.5 Topic Classifier — 主題分類器（RUNTIME: auto）

輸入：`ChunkList`　輸出：`list[TopicCluster]`

**核心流程**：從 chunks 提取候選主題名稱 → 呼叫 `knowledge_base.find_similar_topic()` 查有無語意相似的既有主題 → 有則合併（`is_new=False`），無則新建（`is_new=True`）。

```python
@dataclass
class TopicCluster:
    topic_id: str        # 知識庫 ID（已存在或新建）
    topic_name: str      # 主題名稱，如「Python 遞迴」、「線性代數」
    description: str
    chunk_ids: list[str]
    is_new: bool         # True → 知識庫新建；False → 合併入既有

async def classify(chunks: ChunkList) -> list[TopicCluster]:
    """動態歸類 chunks 到主題，合併語意相似的既有主題"""
```

**相似度判斷**：用 Gemma edge 判斷兩主題是否語意相同（例如「Python list comprehension」與「Python 列表推導式」視為同一主題）。

## 3.6 Knowledge Base — 知識庫（RUNTIME: edge，純邏輯）

SQLite 持久化的主題知識庫，資料跨 session 保留。依賴 `db/models.py` 定義的資料表。

**v1.2 更新**：`MasteryRecord` 從單一 `mastery_level` 升級為**雙軸追蹤**，理解軸與記憶軸分開記錄。

```python
@dataclass
class Topic:
    topic_id: str
    name: str
    description: str
    language: str
    created_at: datetime

@dataclass
class ComprehensionEvent:
    timestamp: datetime
    question_type: str        # explain / apply / analogy / debug
    user_answer: str
    gemma_verdict: str        # no_understanding / partial / solid / deep
    gemma_reasoning: str
    score_delta: float
    is_delayed_test: bool     # 是否為 24hr 延遲測驗

@dataclass
class RetentionEvent:
    timestamp: datetime
    question_type: str        # cloze / multiple_choice / true_false
    response_quality: int     # SM-2：0–5（0=完全忘了，5=秒答）
    new_interval: int
    new_easiness: float

@dataclass
class MasteryRecord:
    concept_id: str

    # ── 理解軸 ──
    comprehension_score: float      # 0.0 ~ 1.0
    comprehension_level: str        # none / surface / deep / transferable
    last_comprehension_test: datetime | None
    pending_confirmation: bool      # True = 當天 deep 回答，等待 24hr 延遲確認
    pending_since: datetime | None  # 待確認標記的時間戳
    comprehension_history: list     # List[ComprehensionEvent]

    # ── 記憶軸 ──
    retention_score: float          # 0.0 ~ 1.0
    sm2_interval: int               # 下次複習距今幾天
    sm2_repetitions: int
    sm2_ease_factor: float
    last_retention_test: datetime | None
    next_review_due: datetime | None
    retention_history: list         # List[RetentionEvent]

    # ── 使用者意圖 ──
    intent: str                     # active / snoozed / archived
    snooze_until: datetime | None   # Snoozed 有到期日時填入

# 分數更新規則
# 理解軸：
#   deep（延遲測驗通過）→ comprehension_level 升一級
#   deep（當天）        → comprehension_score +0.1，設 pending_confirmation = True
#   solid              → comprehension_score +0.05
#   partial            → comprehension_score +0.02
#   no_understanding   → comprehension_score -0.15，標記需重新解釋
# 記憶軸：
#   SM-2 更新 interval / easiness / next_review_due
#   response_quality < 2 → comprehension_score ×= 0.95（輕微連動衰減）

# 公開介面（一律非同步）
async def get_topics() -> list[Topic]
async def find_similar_topic(name: str) -> Topic | None
async def add_topic(name: str, description: str, language: str) -> Topic
async def add_chunks_to_topic(topic_id: str, chunks: ChunkList) -> None
async def get_chunks_by_topic(topic_id: str) -> ChunkList
async def get_mastery(concept_id: str) -> MasteryRecord
async def update_comprehension(concept_id: str, event: ComprehensionEvent) -> MasteryRecord
async def update_retention(concept_id: str, event: RetentionEvent) -> MasteryRecord
async def confirm_delayed(concept_id: str) -> MasteryRecord  # 觸發 24hr 延遲升級
async def set_intent(concept_id: str, intent: str, snooze_until: datetime | None) -> MasteryRecord
```

## 3.7 Quiz Engine — 自適應測驗引擎（RUNTIME: auto）

根據主題熟悉度動態選擇題型，支援三種策略與四種題型。

```python
class QuestionType(Enum):
    SHORT_ANSWER    = "short_answer"     # 簡答題：開放式
    MULTIPLE_CHOICE = "multiple_choice"  # 選擇題：四選一
    TRUE_FALSE      = "true_false"       # 是非題
    FILL_BLANK      = "fill_blank"       # 填空題（新增）

class QuizStrategy(Enum):
    EASY_FIRST = "easy_first"   # 熟悉度低→簡答 中→選擇 高→是非
    HARD_FIRST = "hard_first"   # 熟悉度低→是非 中→選擇 高→簡答
    RANDOM     = "random"       # 隨機選擇題型

@dataclass
class Quiz:
    quiz_id: str
    topic_id: str
    concept: str
    question: str
    question_type: QuestionType
    options: list[str] | None = None  # MCQ 四個選項
    hint: str = ""
    correct_answer: str = ""          # 僅後端使用，不回傳前端

async def generate(
    topic_id: str,
    strategy: QuizStrategy = QuizStrategy.EASY_FIRST,
) -> Quiz

async def evaluate(quiz: Quiz, user_answer: str) -> tuple[float, str]
# 回傳 (score 0.0–1.0, feedback)
```

**題型選擇邏輯**：

| 熟悉度範圍 | EASY_FIRST | HARD_FIRST |
| --- | --- | --- |
| 0.0 – 0.4 | SHORT_ANSWER | TRUE_FALSE |
| 0.4 – 0.7 | MULTIPLE_CHOICE | MULTIPLE_CHOICE |
| 0.7 – 1.0 | TRUE_FALSE | SHORT_ANSWER |

## 3.8 Domain-Aware Quiz Engine — 科目感知出題系統（RUNTIME: auto）

Quiz Engine 在出題前必須先偵測內容的「科目領域（domain）」，再根據 domain 決定最適合的題型與前端互動 UI。這是讓題目與學習內容真正契合的核心機制。

### Domain 分類定義

Gemma 在分析 chunks 時，依以下判斷依據自動標記 domain，每個 topic 只有一個 primary domain：

| Domain | 代表科目 | 自動偵測依據 |
| --- | --- | --- |
| `math_formula` | 機器學習、線代、微積分、統計、物理、化學 | LaTeX 語法、希臘字母密度高、公式佔比 > 20% |
| `programming` | 資料結構、演算法、程式設計、作業系統 | markdown code fence、函式定義、縮排結構 |
| `language` | 英文、語言學、文學、歷史、法律 | 長段落為主、詞彙解釋結構、語言標記 en/zh |
| `memorization` | 生物、解剖學、藥學、歷史年表、地理 | 大量條列清單、時間軸結構、圖片密度高 |
| `calculation` | 會計、財務、物理計算、化學計量 | 數值計算、單位換算、公式帶入數字 |

### 更新後的 Quiz dataclass

新增 `domain` 與各題型專用欄位，前端依此欄位決定渲染哪種互動 UI：

```python
@dataclass
class Quiz:
    quiz_id: str
    topic_id: str
    domain: str               # 新增：math_formula / programming / language /
                              #       memorization / calculation
    question_type: QuestionType
    concept: str
    question: str

    # 各題型 / domain 專用欄位（不適用時為 None）
    formula_tokens: list[str] | None   # math_formula 填空：可點擊的公式元素按鈕
    symbol_palette: list[str] | None   # math_formula 簡答：旁側符號面板
    code_snippet: str | None           # programming：題目附帶的程式碼區塊
    image_ref: str | None              # memorization 圖示標注題的圖片 ID
    sort_items: list[str] | None       # memorization 排序題的待排項目
    match_pairs: list[tuple] | None    # language 配對題的配對組
    numeric_tolerance: float | None    # calculation 數值填空的允許誤差

    options: list[str] | None          # 選擇題 / 填空選項按鈕
    hint: str
    correct_answer: str | list[str]    # 僅後端使用，不回傳前端
```

### Domain × 題型對照表

各 domain 支援的題型與對應前端互動方式：

| Domain | 題型 | 前端互動方式 |
| --- | --- | --- |
| `math_formula` | `fill_blank` | 題目顯示有空格的公式；下方排列 `formula_tokens` 按鈕，點擊填入對應空格 |
| `math_formula` | `short_answer` | 文字輸入框；旁側浮動 `symbol_palette` 符號面板，點擊插入游標位置 |
| `math_formula` | `multiple_choice` | 四個選項按鈕，用於觀念澄清 |
| `programming` | `fill_blank` | Monospace 字體顯示含空格程式碼；`options` 為可點擊 token（`for`、`return`、`None` 等） |
| `programming` | `multiple_choice` | 程式碼閱讀題，問輸出值或時間複雜度 |
| `programming` | `true_false` | 快速確認概念邊界 |
| `language` | `fill_blank` | 句子挖空，`options` 為 3–4 個單字選項按鈕 |
| `language` | `multiple_choice` | 定義選擇、閱讀理解 |
| `language` | `match_pairs` | 術語與定義配對，拖拉或點擊連線（新題型，前端需實作） |
| `memorization` | `true_false` | 快速是非連發（一次 5 題，最後統一給回饋） |
| `memorization` | `sort_items` | 排序題，拖拉排出正確順序（新題型，前端需實作） |
| `memorization` | `image_label` | 圖片標號對應名稱，`options` 為可點擊的名稱按鈕（新題型，需 `image_ref`） |
| `calculation` | `fill_blank` | 數字輸入框，顯示單位提示；`numeric_tolerance` 定義允許誤差 |
| `calculation` | `multiple_choice` | 步驟選擇題，測試推導邏輯而非結果 |

### Gemma Prompt 設計方向

Gemma 出題時的 prompt 必須包含三個步驟，缺一不可：

1. **Domain 偵測**：分析 chunk 內容，輸出 `domain` 標記
2. **題型選擇**：根據 domain + 熟悉度（`mastery_level`）+ 策略（`QuizStrategy`）選定 `question_type`
3. **完整題目生成**：產出 Quiz dataclass 所有必填欄位，包括 `formula_tokens`、`symbol_palette`、`code_snippet` 等 domain 專用欄位

Gemma 回傳格式必須是嚴格的 JSON，不得有多餘的說明文字，後端直接 parse 後對應到 Quiz dataclass。

### 新題型前端實作清單

以下題型在原有四種之外，前端需額外實作：

- `match_pairs`：術語配對，需實作拖拉連線或點擊配對 UI
- `sort_items`：排序拖拉，需實作可拖拉的項目列表
- `image_label`：圖片標號，需實作圖片上標號 + 選項按鈕 UI
- 快速是非連發（`true_false` × 5）：需實作批次模式，答完才顯示回饋

## 4.4 ChatGPT / Gemini 對話來源規範

### 對話取得策略

RecallMap 支援兩種互補的對話輸入方式，禁止以任何形式爬取使用者帳號的完整歷史紀錄（詳見下方「❌ 全帳號爬取」說明）。

**方式一：官方匯出 JSON（歷史舊對話，一次性匯入）**

適用對象：ChatGPT、Gemini（Google Takeout）

使用時機：第一次使用 RecallMap，把過去所有學習對話一次性建立知識庫

ChatGPT 匯出步驟（Onboarding 引導文字）：

1. 前往 ChatGPT 設定 → Data Controls → Export Data
2. 點擊 Export，等待 email 通知（約 5–10 分鐘）
3. 下載 ZIP 並解壓縮
4. 上傳其中的 `conversations.json`

Gemini 匯出步驟（Onboarding 引導文字）：

1. 前往 [takeout.google.com](http://takeout.google.com)
2. 選擇「Gemini Apps 活動」
3. 匯出並下載解壓縮後的檔案

**方式二：分享連結爬取（單篇新對話，日常即時補充）**

適用對象：ChatGPT（Gemini 目前無穩定分享連結功能）

使用時機：上完課後，把今天和 ChatGPT 的對話分享連結貼入，立刻補充進知識庫

URL 格式驗證（後端必須先驗證，不符合則拒絕並回傳明確錯誤訊息）：

```python
import re
CHATGPT_SHARE_PATTERN = re.compile(
    r'^https://chatgpt\.com/share/[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$'
)

def validate_share_url(url: str) -> bool:
    return bool(CHATGPT_SHARE_PATTERN.match(url))
```

爬取實作（使用 Playwright 無頭瀏覽器，後端執行）：

```python
# backend/ingest/chatgpt_share_parser.py
# RUNTIME: edge（本機執行，對話內容不上傳第三方）
from playwright.async_api import async_playwright
import json

async def process(share_url: str) -> list[DocumentChunk]:
    """
    從 ChatGPT 分享連結爬取對話內容。
    頁面為 Next.js SSR，對話資料夾在 __NEXT_DATA__ script tag 的 JSON 中。
    直接解析 JSON 比 DOM 解析更穩定，不易因 UI 改版失效。
    """
    if not validate_share_url(share_url):
        raise ValueError(f"不符合 ChatGPT 分享連結格式：{share_url}")

    async with async_playwright() as p:
        browser = await p.chromium.launch(headless=True)
        page = await browser.new_page()
        await page.goto(share_url, wait_until="networkidle")

        raw = await page.evaluate(
            "JSON.parse(document.getElementById('__NEXT_DATA__').textContent)"
        )
        await browser.close()

    # 對話內容路徑（依 ChatGPT 版本可能變動，需定期確認）
    messages = raw["props"]["pageProps"]["serverResponse"]["data"]["linear_conversation"]
    return _convert_to_chunks(messages, source_url=share_url)
```

注意事項：

- Playwright 需在後端安裝 Chromium（約 300–500 MB），Hackathon 環境請確認記憶體充足
- `__NEXT_DATA__` 的資料路徑可能因 ChatGPT 改版而變動，需定期確認並更新 parser
- **Gemini 一般帳號有分享連結功能**（`gemini.google.com/share/...`）；學校或企業 Workspace 管理帳號可能因管理員政策停用此功能。前端 UI 需在輸入框下方加註：「如果你是用學校帳號登入 Gemini，可能看不到分享按鈕，請改用 Google Takeout 匯出。」
- 前端 URL 輸入框需即時驗證格式，不符合時顯示「這不像是 ChatGPT 的分享連結」

**❌ 全帳號歷史紀錄爬取 — 絕對禁止，且技術上不可行**

有人可能想到：直接用帳密登入後爬取使用者所有的對話歷史。此方式明確禁止，原因如下：

技術面：ChatGPT 和 Gemini 的對話歷史需登入後存取，平台有 CSRF token、Session cookie、裝置信任機制和異常登入偵測。即使取得帳密，也幾乎必然觸發 2FA 或封鎖。

法律面（三重風險）：

- 美國 CFAA（電腦詐騙與濫用法）：未經平台授權存取其伺服器屬聯邦罪，即使使用者自己提供帳密，授權主體仍是平台而非使用者
- 台灣刑法第 358 條：無故輸入他人帳號密碼存取電腦，有刑事責任
- OpenAI / Google ToS：明確禁止自動化存取其服務，違反後帳號可能遭封鎖，且可收到 C&D（停止函）

結論：此方式在技術、法律、合規三個層面均不可行，任何協作者不得在任何版本的程式碼中實作此功能。

# 9. 錯誤處理規範

所有對外 API 呼叫必須有明確的錯誤處理，不得讓例外直接拋到使用者介面。

```python
# backend/engine/gemma_client.py

class GemmaError(Exception):
    """所有 Gemma 相關錯誤的基礎類別"""

class GemmaEdgeUnavailable(GemmaError):
    """本地 Ollama 不可用，已自動 fallback 到雲端"""

class GemmaCloudError(GemmaError):
    """雲端模型呼叫失敗"""

# 錯誤處理模式
async def generate(self, prompt, mode="auto"):
    try:
        if mode in ("auto", "edge"):
            return await self._call_edge(prompt)
    except OllamaConnectionError:
        if mode == "edge":
            raise GemmaEdgeUnavailable("本地模型不可用")
        # auto 模式：靜默 fallback 到雲端
        logger.warning("Edge unavailable, falling back to cloud")
    return await self._call_cloud(prompt)
```

# 11. 跨平台架構

## 11.1 平台策略

RecallMap 採用 **Monorepo** 架構，四個目標平台共用最大化程式碼。

| 平台 | 技術 | 說明 |
| --- | --- | --- |
| Web | Expo web 模式 | 與 mobile 同一份程式碼輸出 |
| iOS | Expo + EAS Build | React Native 編譯 |
| Android | Expo + EAS Build | React Native 編譯 |
| 桌面（Win/Mac） | Tauri | 包裝 Expo web build 為原生應用 |

## 11.2 程式碼共用層次

```
packages/shared   → TypeScript 型別 + API client（所有平台共用）
apps/mobile       → Expo，一份 code 同時跑 iOS + Android + Web
apps/desktop      → Tauri 殼層，包裝 mobile 的 web build
```

**禁止**在各 app 內自行定義重複型別，所有共用型別必須放在 `packages/shared/src/types/`。

## 11.3 開發優先順序

1. **Web**（Expo web 模式）— 主要開發與驗證目標
2. **Mobile**（同一份 Expo code，針對 RN 元件微調）
3. **Desktop**（Tauri 包 web build，加本地 Ollama 整合）

## 11.4 Tauri 桌面版特有整合

桌面版因可直接與本機溝通，額外支援：

- 直接啟動 / 監控 Ollama 進程（真正 Privacy-First 離線）
- 本機檔案系統直接存取（不需上傳）
- 無需網路即可完整運行
